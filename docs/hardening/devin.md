# Hardening Devin

Devin is Cognition's autonomous AI software engineer. Unlike Claude Code (which runs on your machine), Devin runs in a cloud VM managed by Cognition. This fundamentally changes the security model: you're trusting a third party with your code, credentials, and infrastructure access.

This guide covers what Devin protects, what it doesn't, and what you can do about the gaps.

**Time needed:** ~30 minutes to review and configure. Less hands-on than Claude Code hardening because most controls are in Cognition's web UI, not local config files.

## Security Model Overview

Devin operates in a cloud-hosted sandboxed compute environment with shell access, a code editor, and a browser. Each session runs on its own isolated machine.

| Property | Devin | Claude Code |
|----------|-------|-------------|
| **Execution environment** | Cloud VM (Cognition-managed) | Local machine (your hardware) |
| **Sandbox** | Container isolation per session | Seatbelt (macOS) / bubblewrap (Linux) |
| **Secret management** | Built-in Secrets Vault (web UI) | None built-in (env vars, credential managers) |
| **Hooks / policy enforcement** | No user-defined hooks | Full hook system (PreToolUse/PostToolUse) |
| **Permission model** | No per-action approval gate | Interactive approval, auto mode, or bypass |
| **Audit logging** | Enterprise API audit logs | Local JSONL via hooks |
| **Network control** | Enterprise VPC deployment | Sandbox network allowlisting |
| **Code access** | Cognition sees your code | Code stays local |
| **MCP support** | No | Yes |

**The core tradeoff:** Devin's cloud model gives you ephemeral VMs and session isolation for free, but you're trusting Cognition with your source code and credentials. Claude Code keeps everything local but requires more manual hardening.

## What Devin Gets Right

### Session isolation

Each Devin session runs on its own isolated machine. When the session ends, the VM is destroyed. This is genuinely better than Claude Code's default model, where multiple sessions share your host filesystem, environment variables, and network.

There's no equivalent of the "one session reads credentials that another session left in the environment" problem that exists with Claude Code worktrees.

### Secrets Vault

Devin provides a built-in Secrets Vault accessible through the Settings page in the web UI. It functions similarly to GitHub Actions secrets or Vercel Environment Variables, but scoped to Devin's cloud workspaces. ([Devin docs](https://docs.devin.ai/enterprise/security-access/security/enterprise-security))

How it works:
- You add secrets through the web UI (Settings > Secrets)
- Secrets are injected into sessions as environment variables
- Secrets are not exposed in the chat interface
- You can share secrets with Devin in bulk

This is better than `.env` files (which Claude Code auto-loads into its context) because Devin's vault at least separates secret storage from the conversation.

### SOC 2 Type II certification

Cognition obtained SOC 2 Type II certification in September 2024. All data transmission is encrypted in transit and at rest. Production systems are continuously monitored through logging and real-time dashboards. ([Trust Center](https://trust.cognition.ai/))

### Data ownership

By default, Cognition does not train its models on customer data or code. Output generated by Devin (code, work product, etc.) is the customer's intellectual property. This is documented in their enterprise security docs.

## Known Vulnerabilities

Devin has a documented history of security issues that you should factor into your risk assessment.

### CVE-2024-56083: VSCode Live Share URL Exposure

Before December 12, 2024, anyone who discovered the `https://vscode-randomly_generated_string.devinapps.com` URL for a "Use Devin's Machine" session could gain **write access** to the code in that session. ([CVE details](https://www.cvedetails.com/cve/CVE-2024-56083/))

The URL could be discovered if a user posted a screenshot of a Devin session to social media or publicly livestreamed their session. This was fixed, but it demonstrated that Devin's session isolation relied partly on URL obscurity rather than authentication.

### Prompt Injection: Complete Secret Exfiltration

Johann Rehberger of Embrace The Red spent $500 testing Devin for prompt injection vulnerabilities and found it "completely defenseless." ([Embrace The Red](https://embracethered.com/blog/posts/2025/devin-i-spent-usd500-to-hack-devin/))

**Exfiltration vectors confirmed:**

1. **Shell tool**: Devin runs `curl`, `wget`, or Python scripts that send data to attacker-controlled servers. All secrets available as environment variables are accessible. ([Details](https://embracethered.com/blog/posts/2025/devin-can-leak-your-secrets/))

2. **Browser tool**: Devin navigates to an attacker-controlled URL with sensitive data appended as query parameters.

3. **Image rendering**: Devin renders images from untrusted domains, allowing attackers to append sensitive data to image URLs for exfiltration.

4. **Malicious instructions in GitHub issues/websites**: By planting prompt injection payloads in content Devin processes (GitHub issues, websites, documentation), attackers can instruct Devin to exfiltrate secrets, download malware, or expose network ports to the internet.

**The attack chain:**
1. Attacker plants malicious instructions in a GitHub issue
2. User asks Devin to investigate the issue
3. Devin reads the instructions and follows them
4. Devin exfiltrates Jira tokens, Slack tokens, and other secrets to attacker-controlled servers

Rehberger reported these to Cognition on April 6, 2025. Cognition acknowledged receipt but provided no fix timeline. As of publication (120+ days later), the status of fixes was unclear. ([Disclosure](https://embracethered.com/blog/posts/2025/devin-can-leak-your-secrets/))

### AI Kill Chain: Port Exposure

Devin can be tricked via prompt injection to expose ports to the internet, creating a remote shell accessible to attackers. This turns Devin into what Rehberger calls a "ZombAI" -- a remote-controlled agent. ([Kill chain writeup](https://embracethered.com/blog/posts/2025/devin-ai-kill-chain-exposing-ports/))

### Pillar Security Assessment

Pillar Security published a broader assessment of SWE agent risks, specifically calling out Devin and similar agents for lacking adequate input validation, output filtering, and action authorization. ([Pillar Security](https://www.pillar.security/blog/the-hidden-security-risks-of-swe-agents-like-openai-codex-and-devin-ai))

## Secrets Vault: Limitations

While the Secrets Vault is better than `.env` files, it has significant limitations:

### No per-environment scoping

All secrets are available to all sessions. There's no way to say "this secret is only for production" or "this secret is only for the staging environment." Every Devin session gets every secret.

This means a session investigating a bug in a test repo has the same access to your production database credentials as a session deploying to production.

### Secrets are environment variables in the VM

Once injected, secrets are environment variables inside the session VM. Any process running in the VM can read them. Since Devin can run arbitrary shell commands, a prompt injection attack has full access to all secrets.

### No rotation or expiry

The Secrets Vault is a key-value store, not a credential manager. There's no automatic rotation, no expiry dates, no access logging at the individual-secret level.

### No hook-based protection

Unlike Claude Code, Devin has no user-configurable hooks to intercept tool calls. You can't install a "secret guard" that blocks writes containing credential patterns. You can't install a "bash firewall" that blocks exfiltration commands. Cognition controls the execution environment, and user-defined security policies are not supported.

## Hardening Steps

Given the limitations above, here's what you can do:

### 1. Minimize secrets exposure

**Only add secrets Devin actually needs.** If Devin is writing code that interacts with your database, it probably needs a database URL. It does not need your AWS root credentials, your Stripe live keys, or your production OAuth tokens.

Create a minimal set of secrets per project:
- Use read-only database credentials where possible
- Use API keys with minimal scopes (read-only, specific resources)
- Never give Devin production credentials if staging credentials will do

### 2. Use fine-grained access tokens

For SCM (GitHub/GitLab) integration:
- Use fine-grained Personal Access Tokens, not classic tokens
- Scope to specific repositories Devin needs access to
- Set expiration dates (30-90 days)
- Review and revoke regularly

The administrator installing an integration can review and manage all permissions granted to Devin. Depending on integration type, permissions can be adjusted at any time via the SCM platform's settings.

### 3. Enterprise VPC deployment

For organizations with sensitive code, Devin Enterprise supports deployment within your Virtual Private Cloud:

- **AWS VPC**: Development boxes deployed within your VPC. Data stays in your controlled environment. ([AWS setup guide](https://docs.devin.ai/enterprise/vpc/aws-setup))
- **Private Link**: VPC connects to Cognition's infrastructure via AWS Private Link or IPSec tunnel
- **Single-tenant**: Customer Dedicated SaaS model with auto-scaling, customer-isolated environment

This is the most meaningful hardening available for Devin: if the VM runs in your VPC, you control network egress and can apply your own firewall rules.

**Cost:** Enterprise VPC is a paid enterprise feature. It's not available on Devin's standard plans.

### 4. Restrict repository access

When configuring the SCM integration, select only the repositories Devin needs. Don't give Devin access to all repositories in your organization.

### 5. Monitor with audit logs

Devin Enterprise provides audit logging via API:

```bash
# Fetch recent audit logs (Enterprise API)
curl -H "Authorization: Bearer $DEVIN_API_TOKEN" \
  https://api.devin.ai/v3/enterprise/audit-logs
```

The audit logging system tracks session creation, member management, search queries, and AI guardrail violations. ([Audit logs docs](https://docs.devin.ai/enterprise-api/audit-logs))

**Limitation:** Audit logs are an enterprise feature. Standard plan users have limited visibility into what Devin does during a session.

### 6. Don't share session URLs

CVE-2024-56083 was patched, but the principle remains: Devin session URLs should be treated as sensitive. Don't post screenshots containing session URLs. Don't stream Devin sessions publicly.

### 7. Review Devin's output before merging

Devin creates pull requests. Always review them as carefully as you would review a human contributor's PR. Specifically:

- Check for hardcoded credentials (Devin might embed secrets it has access to)
- Check for new network calls to unfamiliar domains
- Check for new dependencies (supply chain risk)
- Check for permission changes in CI/CD configs

### 8. Be cautious with untrusted inputs

The prompt injection research shows Devin will follow malicious instructions embedded in GitHub issues, websites, and documentation. Don't point Devin at:

- Issues filed by unknown contributors without reviewing them first
- Arbitrary URLs from untrusted sources
- Repositories you haven't reviewed

## What Devin Cannot Do (That Claude Code Can)

| Capability | Claude Code | Devin |
|-----------|-------------|-------|
| User-defined hooks (PreToolUse/PostToolUse) | Yes | No |
| Block specific commands before execution | Yes (bash firewall hook) | No |
| Block secret writes to files | Yes (secret guard hook) | No |
| Custom audit logging | Yes (audit logger hook) | Enterprise API only |
| Network allowlisting (user-configured) | Yes (sandbox settings) | Enterprise VPC only |
| Per-action permission approval | Yes (interactive mode) | No |
| Auto mode (classifier-based approval) | Yes | No |
| Local execution (code stays on your machine) | Yes | No |
| MCP server integration | Yes | No |

This is the fundamental gap: Devin gives you no programmatic policy enforcement layer. Cognition controls the execution environment, and you rely on their security measures rather than implementing your own.

## Security Comparison: Honest Assessment

### Where Devin is stronger

- **Session isolation**: Each session gets a fresh VM. This is strictly better than Claude Code's worktree model for isolation.
- **No local credential exposure**: Devin can't read your `~/.ssh`, `~/.aws`, or `~/.zshrc`. It only sees what you explicitly put in the Secrets Vault.
- **Ephemeral by default**: When a session ends, the VM is destroyed. There's no persistent state that could leak between sessions.
- **No sandbox escape risk**: The "sandbox" is a full VM. There's no symlink-following CVE equivalent because the VM boundary is the isolation boundary.

### Where Claude Code is stronger

- **User-defined security policies**: Hooks let you enforce arbitrary policy. You can block, audit, and modify any tool call.
- **Transparency**: You can see exactly what Claude Code does via hooks and audit logs. Devin's session internals are opaque.
- **Code stays local**: Your source code never leaves your machine. With Devin, Cognition has access to your code.
- **No third-party trust**: You don't need to trust Anthropic's infrastructure with your code or credentials (only with your prompts via the API).
- **Network control**: You configure the sandbox network allowlist yourself. With Devin, you need Enterprise VPC to control egress.

### Where both are weak

- **Prompt injection**: Both Claude Code and Devin are vulnerable to indirect prompt injection from untrusted inputs. Neither has robust input validation that prevents malicious instructions embedded in code, issues, or documentation from being followed.
- **Secrets in process memory**: Both have the same fundamental problem -- once a secret is available to the agent (via env var or Secrets Vault), a prompt injection can exfiltrate it.
- **No formal verification**: Neither tool has formally verified security properties. Both rely on best-effort sandboxing and policy enforcement.

## Deployment Decision Framework

| Scenario | Recommendation |
|----------|---------------|
| Solo developer, all code is open source | Either works. Claude Code gives more control. |
| Solo developer, private code, no sensitive credentials | Either works. Devin's isolation is simpler to set up. |
| Working with production credentials | Claude Code with container isolation. Don't put production creds in Devin's Vault. |
| Auditing untrusted code | Claude Code in Trail of Bits devcontainer. Devin has no equivalent containment. |
| Enterprise team with compliance requirements | Devin Enterprise VPC or Claude Code in your own infrastructure. |
| Processing untrusted GitHub issues | Neither is safe without human review of the issue content first. |
